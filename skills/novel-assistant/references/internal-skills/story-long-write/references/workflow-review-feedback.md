# workflow-review-feedback.md：阶段审阅反馈工作流

本文件处理“用户正在审阅某个阶段产物并提出意见”的场景。它不是正文回炉，也不是默认的 Revision Impact Analysis。

---

## 适用条件

命中任一条件即优先进入本流程：

- prompt 带有前端或工作台注入的 `【阶段上下文】`、`当前阶段`、`当前审阅文件`、`只处理当前阶段相关问题`。
- 用户正在评价当前大纲、卷纲、细纲或正文产物，例如“进度太快”“我不认同”“这个出现太早”“需要增加过渡”“语言学习要循序渐进”。
- 用户没有明确要求“立即批量改正文 / 重排章节文件 / 执行回炉复检”。

硬规则：

1. 不得直接进入 Revision Impact Analysis。
2. 不得把阶段反馈升级成 20-30 章重排、600+ 文件影响分析或全文大修。
3. 不得用新方案覆盖用户刚否定的理解；先记录“用户明确否定了什么”。
4. 只处理当前阶段相关问题；当前阶段外的影响先写入“后续影响”。如果用户提出的是全局级 / 大纲级修改，必须先做可行性验证；验证通过后立即实施到当前阶段和必要的下游大纲产物。

---

## Step 1：用户反馈摄取

先从用户最新消息中抽取反馈，不急着给大方案。输出不超过 6 条，格式如下：

```md
我听到的是：
- 硬约束：
- 明确否定：
- 希望加强：
- 希望后移：
- 当前阶段范围：
- 可能影响后续：
```

区分三类内容：

- **硬约束**：用户已经明确要求必须遵守，例如“御兽宗后移”“兽语学习不能一上来会”“进度太快要增加过渡”。
- **明确否定**：用户说“不认同”的解释，例如“灵石认知不是单独学习模块，而是生存、语言、交流之后才出现的认知结果”。
- **可商量偏好**：用户只表达方向但未锁死细节，例如“山脉成长更充实”“挑战难度增加”。

如果可以从用户话里确定方向，不要反复问 Q1-Q4。只有真正无法判断的关键点才问 1 个当前阶段问题。

---

## Step 2：阶段范围判定

根据 `当前阶段` 和 `当前审阅文件` 限定可处理产物：

| 当前阶段 | 默认可处理产物 | 不得自动处理 |
|---|---|---|
| 市场定位 / 选题 | `市场定位设定/`、`选题决策.md`、`设定/题材定位.md` | 正文、细纲批量改写 |
| 核心承诺 / 核心设定 | `核心承诺设定/`、`设定/`、`追踪/主线承诺.md` | 正文重写 |
| 剧情引擎 / 世界规则 | `剧情引擎设定/`、`设定/世界观/`、`追踪/主线承诺.md` | 章节正文 |
| 总纲与卷纲 | `大纲/大纲.md`、`大纲/总纲.md`、`大纲/第X卷/卷纲.md`、旧项目 `大纲/卷纲_第*.md`、`追踪/主线承诺.md`、`追踪/审阅反馈.md` | 已写正文、批量重排章节文件 |
| 章节细纲 | `大纲/第X卷/细纲_第XXX章.md`、旧项目 `大纲/细纲_第XXX章.md`、`追踪/章节契约/`、`追踪/审阅反馈.md` | 正文重写，除非用户明确要求 |
| 正文生产 | `正文/第X卷/第XXX章_*.md`、旧项目 `正文/第XXX章_*.md`、`追踪/章节契约/`、`追踪/交接包/`、`追踪/审阅反馈.md` | 跨卷重构，除非进入 RIA |

如果阶段是“总纲与卷纲”，只能先修订宏观节奏、认知阶梯、势力登场顺序和卷内功能，不要直接发散成每章正文改写。

### 反馈影响级别

完成阶段范围判定后，必须把反馈归入一个且仅一个最高影响级别：

| 影响级别 | 典型变化 | 返回节点 |
|---|---|---|
| `prose` | 措辞、文风、局部表达，不改变章节事实 | `prose` |
| `brief` | 本章目标、场景、阻力、信息或钩子 | `chapter_brief` |
| `detail_outline` | 阶段事件、章节结构、插章、合并、删章、扩容或缩容 | `stage_detail_outline` |
| `volume_outline` | 本卷目标、卷内节奏、卷尾状态或跨卷接口 | `volume_outline` |
| `master_outline` | 全书主线、核心承诺、成长终局或总纲结构 | `master_outline` |

多项反馈同时出现时取最高影响级别，不得因用户只点名一章就把总纲级变化降为正文级。可调用 `scripts/lib/lifecycle-impact.js` 的 `classifyFeedbackImpact(feedback)` 统一分类；分类只决定回退层级，不授权修改资产。

### 生命周期失效传播

上游资产变化后，先调用 `invalidateDownstream(graph, changedAsset)` 生成影响结果，再决定本阶段可执行内容。只允许写生命周期 metadata：

- 已接受的下游规划资产标记 `needs_recheck`。
- 尚未接受且依赖已变化上游的下游资产标记 `invalidated`。
- 已接受正文标记 `preserve_until_proven_invalid`，正文原文和 accepted revision 保持不变，直到独立影响分析证明必须回炉。

禁止根据反馈影响链删除、清空、覆盖或批量重写 accepted prose。正文删除、覆盖和批量回炉必须另建高风险事务，明确影响范围、快照、用户确认与复检证据。

---

## Step 3：反馈落盘

默认将本次反馈追加到 `追踪/审阅反馈.md`。如果文件或目录不存在，先创建目录和文件。写入模板：

```md
## {YYYY-MM-DD HH:mm} - {当前阶段} - {当前审阅文件}

### 用户原话摘要
- 

### 已锁定约束
- 

### 明确否定
- 

### 当前阶段处理范围
- 

### 后续阶段影响
- 
```

落盘时保留用户核心表述，不要改写成模型自己的设定。每次回复都要告诉用户本轮反馈写入了哪个项目相对路径。

---

## Step 4：可行性验证

当用户提出全局级 / 大纲级修改建议（例如“整体进度太快”“某势力登场后移”“山脉成长要丰富”“原有细纲章节后移”“后面需要回收钩子”），不要只停在建议。先验证它是否能执行。

输出并实际检查：

```md
### 可行性验证
- 核心承诺：是否仍服务本书核心爽点/卖点？
- 因果链：前因、成长、认知、交流、资源理解是否顺序成立？
- 读者期待：延缓登场是否增加期待，而不是拖沓？
- 章节容量：当前卷/前 10 章是否有空间承接新增过渡？
- 下游影响：哪些卷纲、细纲、伏笔、钩子回收需要同步？
- 风险结论：通过 / 需补充信息 / 不建议执行
```

判定规则：

- **通过**：不违背核心承诺，因果链更清楚，读者期待更强，影响面可在大纲/细纲内同步。验证通过后立即实施，不再反复问用户“要不要改”。
- **需补充信息**：只差一个关键选择，例如“御兽宗最早出现在第几卷”。最多问 1 个问题。
- **不建议执行**：会削弱核心卖点、破坏主线承诺或造成已写正文大面积重写。说明原因，并给出替代方案。

### 大纲更新事务门禁

凡是“更新大纲 / 更新卷纲 / 成长进度设定对齐 / 设定基准对齐 / Phase F 更新”这类全局级或卷级大纲变更，在验证通过后也不能马上对目标文件执行 Update/Edit。必须先建立事务边界，避免出现读 1 个文件后直接 Update、hook 报错后继续硬改、半更新停在输入框的情况。

执行前先输出并落盘 `Update Scope Contract` 到 `追踪/审阅反馈.md` 或 `追踪/大纲更新事务_{YYYYMMDD_HHMMSS}.md`：

```md
### Update Scope Contract
- 用户原始意图：
- 修改类型：设定基准对齐 / 节奏重排 / 章节扩容 / 缩容 / 合并 / 局部修补
- 目标权威设定：
- 先列受影响文件：
- 不允许修改：
- dry-run 差异计划：
- 验收命令：
- 回滚依据：
```

硬规则：

1. **先列受影响文件**：至少列出总纲/卷纲/细纲/追踪文件；不确定时先 `rg` 定位旧术语（例如 `元婴崛起`、旧日期、旧成长进度设定、旧权威设定）。
2. **dry-run 差异计划**：先写“旧值 -> 新值”的差异表，确认会改哪些段落，再进入编辑；不得读 1 个文件后直接 Update。
3. **版本快照**：运行或提示运行 `node scripts/chapter-assets-build.js <book-dir> --write` 与 `node scripts/story-version-snapshot.js <book-dir> --reason "outline-ssot-update" --files "<受影响文件>" --write`。快照失败时停止，不继续编辑。
4. **分文件事务**：每个文件修改后立即读取关键段落复核；不要连续十几次 Update 同一大纲文件后才检查。
5. **设定基准词表**：涉及成长进度、境界曲线、时间轴、势力登场顺序时，建立旧词/新词映射，禁止出现旧权威设定和新权威设定同文件混杂。只有项目事实明确为修真/仙侠时，才把“修真进度阈值”作为设定基准示例；其他题材必须改写为对应的成长规则或能力边界。
6. **收束报告**：结束前写明已改文件、未改文件、阻塞项、下一步；不能只停在工具输出或 task list。

### Hook P0 收束门禁

如果编辑后出现 `PostToolUse:Edit hook`、`AI 痕迹检测`、`P0 痕迹`、`output pollution`、长术语重复填充或正文/报告污染提示，即使 hook 是 non-blocking，也按硬阻塞处理。

处理规则：

1. **P0 是硬阻塞**：P0 命中后不得继续编辑同一目标文件，不得把当前文件视为已完成。
2. **先读回文件**：立即读取刚改文件的相关段落，定位污染来源和是否已经部分落盘。
3. **运行污染检查**：对报告、修复方案、大纲更新说明、审阅反馈等长文本，运行 `node scripts/output-pollution-check.js --learn --project-root <book-dir> <目标文件>`；正文另走 `story-prose-gate.js` 与 `check-ai-patterns.js`。
4. **回退或修补污染段**：只修污染段和本次更新段；无法可靠修复时，写入 `paused_after_hook_p0` 到事务文件，列出目标文件、hook 输出、最后成功步骤和恢复命令。
5. **复扫到通过**：P0 清零后才继续下一文件；复扫失败时停止并收束，不继续消耗长上下文硬改。

### 可见长回复污染门禁

阶段反馈、可行性验证、设定基准对齐建议、大纲更新计划或修复方案超过 800 中文字符时，不得直接把长段落输出给用户。直接回复用户前，先把拟回复内容写入 `追踪/.visible_reply_draft_{YYYYMMDD_HHMMSS}.md` 或本轮事务文件的 draft 段，运行 `node scripts/output-pollution-check.js --learn --project-root <book-dir> <draft-file>`。

如果 draft 命中领域术语循环、阶段标签循环、重复填充或已学习污染词组：

1. 不得把污染文本继续输出到聊天窗口。
2. 删除污染段，回到事实点，用短句/表格重写。
3. 复扫到 0 后再输出给用户。
4. 如果污染已经在聊天窗口出现，立即停止生成，落盘 `paused_after_output_pollution`，说明污染短语、最后可信事实点和恢复动作，不要继续补长报告。

如果污染还没有写入 draft，但已经在阶段名、回炉标题、选择项或可见说明里开始重复同一短语 3 次以上，直接按 `blocked_model_degradation` 处理，不得把污染短语扩写成“完整事务说明”。源文件中的污染短语只能标记为 `污染段#N`，证据给路径/行号；污染段本身不作为事实依据。

### 污染恢复协议

阶段反馈污染后不能把污染段当成“草稿继续改”。按下面步骤恢复：

1. **最后可信事实点**：回到用户反馈、已读文件和可验证差异表，标记污染前最后一条可信判断。
2. **丢弃污染段**：污染起点之后的建议、结论、修复动作全部作废，不进入大纲/细纲/追踪。
3. **分块重写**：只按“用户意图 -> 可行性 -> 影响文件 -> 本次只做什么 -> 下一步”五块重写，每块不超过 8 条。
4. **复扫**：每块写入 draft 后运行 `output-pollution-check.js`；通过后才合并。
5. **连续 2 次失败**：停止本轮长回复，写 `paused_after_output_pollution`，建议新会话从最后可信事实点继续，不在当前污染上下文里硬补。

---

## Step 5：全局级 / 大纲级修改实施

验证通过后，按影响范围直接实施。只输出当前阶段和必要下游大纲产物，不要给跨阶段巨型方案。

### 大纲修改类型判定

实施前必须先判定用户要的是哪一类大纲修改。不要把局部细纲修改、扩容插章、缩容删减、章节合并混成一套流程。

| 类型 | 典型意图 | 是否触发章节偏移 | 执行重点 |
|---|---|---|---|
| 局部细纲修改 | “改第 6 章细纲”“这一章加一个钩子”“这章节奏慢一点” | 不触发章节偏移 | 只修改目标细纲；钩子可登记到后续合适章节回收 |
| 扩容插章 | “前期太快，插 3 章过渡”“御兽宗后移，山脉成长扩充” | 触发后续章节后移 | 先偏移后填充，保护原有细纲 |
| 缩容删减 | “这一段太拖，压缩 3 章成 1 章”“删掉两章过场” | 触发后续章节前移 | 先保留关键 beat 和钩子，再删减/压缩，最后前移后续章节 |
| 章节合并 | “第 6-7 章合并”“两个细纲内容重复，合成一章” | 触发后续章节前移 | 合并目标、情绪、钩子和伏笔，删除重复 beat，重写承接 |

判定规则：

- 只改某一章或某几章内部内容，没有新增/删除章节数 → 局部细纲修改。
- 新增章节、插入过渡、把某事件后移但原章节仍保留 → 扩容插章。
- 删除章节、压缩篇幅、减少章节总数 → 缩容删减。
- 多个原章节合成更少章节 → 章节合并。
- 用户表达含糊时，先按“是否改变章节总数”判断；仍不清楚时只问 1 个问题。

扩容、缩容、插章、合并和删章都属于结构变更。无论当前在正文、Brief 还是审阅节点，必须先调用 `buildReplanActions(impact)` 返回 `stage_detail_outline` 或更高计划层，完成影响分析后才能执行。回退包至少包含：

- 原章节号、章节名、核心事件、章尾钩子和待回收伏笔清单。
- 可复用正文、Brief、细纲片段及其来源 revision；默认保留，不得以“结构已变”为由覆盖。
- 章节移动、合并、保留、待验证和新增缺口的映射表。
- `preserve_chapter_names=true` 与 `preserve_reusable_content=true`；只有用户明确改名或新核心事件与旧标题冲突时才提出改名。
- 对受影响下游资产的 `needs_recheck`、`invalidated`、`preserve_until_proven_invalid` metadata 计划。

对用户示例“疯了吧？老子都杀疯了，你才契约我？”应这样理解：

- 早期主轴是生存本能，不是学习课程。
- 兽语/语言能力应在等级、身体经验和环境压力累积后解锁。
- 与人交流发生在语言能力之后，认知升级发生在交流之后。
- 灵石认知不是独立学习模块，而是“生存经验 → 语言 → 交流 → 交易/资源认知”的自然结果。
- 山脉成长段需要更多挑战、试错、受伤、捕猎、逃亡和选择代价。
- 御兽宗出现过早会压缩野外成长，应后移到主角具备基础语言/交流/认知之后。

实施顺序：

1. **升级迁移门禁**：如果项目仍是旧扁平结构，先执行内部模块 `story-setup` 的迁移步骤（用户提示统一写 `/novel-assistant 准备写书`），或运行 `node scripts/story-project-migrate.js <book-project-dir> --write`，再刷新 `chapter-assets-build.js` 和 schema。不要在旧扁平结构上继续做大规模偏移。
2. **版本快照**：运行 `node scripts/chapter-assets-build.js <book-project-dir> --write`，再用 `node scripts/story-version-snapshot.js <book-project-dir> --reason "<本次修改原因>" --files "<受影响文件列表>" --write` 保存快照。
3. **全局大纲同步**：修改 `大纲/大纲.md` 或 `大纲/总纲.md`，把新的成长阶梯、势力登场顺序、核心节奏写成全书规则。
4. **卷纲同步**：修改 `大纲/第X卷/卷纲.md`（旧项目回退 `大纲/卷纲_第X卷.md`），明确本卷新增过渡、挑战升级、势力后移和阶段结尾状态。
5. **先偏移后填充**：如果需要插入过渡章或把某剧情后移，先冻结原有细纲并建立章节偏移映射表，再把受影响的原有细纲整体后移，最后再填充中间缺口。不得先写新过渡章再临时挪后文，避免覆盖旧细纲或留下空号。
6. **细纲同步**：修改受影响的 `大纲/第X卷/细纲_第XXX章.md`。如果章节后移，更新本章核心事件、章尾钩子和下一章承接；不要只改标题。用户确认过的章节名默认保留，除非用户明确要求改名或标题与新核心事件冲突。
7. **钩子回收同步**：如果后移剧情牵涉伏笔、承诺或钩子回收，更新 `追踪/伏笔.md`、`追踪/主线承诺.md` 或对应细纲的“章尾钩子 / 待回收伏笔”。
8. **资产与 schema 刷新**：修改结束后运行 `node scripts/chapter-assets-build.js <book-project-dir> --write` 和 `node scripts/story-schema-build.js <book-project-dir> --write`。
9. **细纲连续性校验**：完成偏移和填充后，检查 `大纲/第X卷/细纲_第XXX章.md` 是否从起始章到结束章连续，不得留下空号；发现缺失章节时先补齐最小功能章纲，再汇报完成。
10. **不碰正文边界**：除非用户明确要求修改已写正文，否则只修改大纲/卷纲/细纲/追踪，不进入正文回炉。

局部细纲修改的执行算法：

1. 只修改目标细纲（优先 `大纲/第X卷/细纲_第XXX章.md`），不触发章节偏移。
2. 如果新增钩子、伏笔或承诺，更新当前细纲的章尾钩子，并写入 `追踪/伏笔.md` 或 `追踪/主线承诺.md`。
3. 钩子可登记到后续合适章节回收，不必为了回收钩子强行插章；优先选择已有后续章节中情绪高潮或信息揭露最匹配的一章。
4. 如果该钩子会改变后续章节核心事件，再标记受影响细纲并做局部同步；否则只登记回收计划。

章节后移的执行算法：

```md
### 章节偏移映射表
| 原章节 | 新章节 | 原核心事件 | 处理 |
|---|---|---|---|
| 第006章 | 第009章 | 御兽宗初现 | 整体后移 |
| 第007章 | 第010章 | 初次交涉 | 整体后移 |

### 新增/填充章节
| 新章节 | 功能 | 承接前章 | 推向后章 |
|---|---|---|---|
| 第006章 | 山脉生存挑战升级 | 承接第005章伤势/饥饿 | 推向第007章语言萌芽 |
| 第007章 | 兽语/语言前置条件 | 承接生存压力 | 推向第008章初步交流 |
| 第008章 | 交流前的误解与试探 | 承接语言萌芽 | 推向第009章御兽宗初现 |
```

实施时按这个顺序：

1. 先冻结原有细纲：记录原章节号、章名、核心事件、章尾钩子和待回收伏笔。
2. 生成章节偏移映射表：明确每个被后移细纲的新章节号。
3. 先整体后移原有细纲：旧内容迁移到新编号，保留原核心事件，但重写承接、章尾钩子和伏笔状态。
4. 再填充中间缺口：新增过渡章必须有功能，不得写“占位”；至少包含核心事件、目标情绪、因果链、章尾钩子。
5. 承接检查：新增章节的结尾必须能推向后移后的旧剧情；后移后的旧章节开头也要承认新增过渡带来的状态变化。
6. 钩子回收检查：任何章尾钩子、伏笔或承诺的回收章节变化，都要同步 `追踪/伏笔.md` 和相关细纲。
7. 细纲连续性校验：按章节号列出当前受影响区间，确认没有缺失章节、重复章节或跳章。

缩容删减的执行算法：

缩容不是简单删除。必须先保留关键 beat、伏笔、钩子和角色状态，再压缩章节，并让后续章节前移。

```md
### 删减映射表
| 原章节 | 处理 | 保留内容 | 去向 |
|---|---|---|---|
| 第006章 | 合并进第005章 | 语言萌芽 beat、受伤代价 | 第005章后半 |
| 第007章 | 删除 | 重复赶路 | 无 |
| 第008章 | 前移到第006章 | 初次误解交流 | 第006章 |
```

执行顺序：

1. 标出被删/被压缩章节的关键 beat、必须钩子、待回收伏笔和角色状态变化。
2. 先把必须保留内容合并到前后最合适细纲；无法承接的内容不得删除，只能改写或移到后续章节。
3. 删除或压缩空转内容后，生成删减映射表。
4. 后续章节前移，更新细纲文件编号、章名、承接、章尾钩子和伏笔回收章节。
5. 运行细纲连续性校验，确认没有缺失章节、重复章节或跳章。

章节合并的执行算法：

合并类修改必须先合并后前移：先把多个原细纲压成一个可承接的新细纲，再让原后续章节前移。不得先删除中间章节再回头补承接。

```md
### 合并映射表
| 原章节组 | 新章节 | 保留关键 beat | 删除/压缩 | 钩子处理 |
|---|---|---|---|---|
| 第006-007章 | 第006章 | 生存挑战、语言萌芽 | 重复赶路 | 章尾钩子改为误解交流 |
```

执行顺序：

1. 读取待合并细纲，抽取每章核心事件、目标情绪、关键 beat、章尾钩子和伏笔。
2. 合并为一个新细纲：只能保留一个主情绪和一个主高潮；副 beat 合入铺垫或过场。
3. 删除重复、同功能、低收益的情节点，保留会影响后续承接的角色状态变化。
4. 原后续章节前移，更新承接和钩子回收。
5. 对合并后的章节做承接检查：前一章能推入，新后一章能接住。

对章节后移的最低要求：

- 原章节留下什么替代事件，不能出现空洞。
- 后移后的首次登场要更强，必须说明期待如何累积。
- 被后移章节前后的章尾钩子要重新承接。
- 相关细纲必须更新因果链，而不是只把章节号整体挪动。
- 后移不能造成原有章节缺失；如果插入 3 章过渡，原第 N 章到第 M 章必须整体顺延 3 章，并逐章检查承接。

对钩子回收的最低要求：

- 新增/后移的钩子写入 `追踪/伏笔.md`。
- 每个钩子至少标明：埋设章节、预期回收章节、当前状态、读者期待功能。
- 如果回收章节变化，更新对应 `大纲/第X卷/细纲_第XXX章.md` 的高潮或章尾承接。

推荐输出结构：

```md
### 已记录的修订约束
- 

### 可行性验证
- 

### 已实施
- 

### 暂不实施
- 

### 影响产物
- `追踪/审阅反馈.md`
- `{当前审阅文件}`
- `大纲/第X卷/细纲_第XXX章.md`
- `追踪/伏笔.md`
```

如果验证通过且相关文件存在，直接编辑文件并说明改了哪些项目相对路径；如果文件缺失，先说明缺失文件并创建当前阶段允许的新文件或给出最小补齐动作。

## Step 6：何时升级到 RIA

只有以下情况才进入 `revision-impact-analysis.md`：

- 用户明确要求改已写正文。
- 用户要求批量重排章节编号或移动大量章节。
- 用户要求删除/提前回收伏笔，且会影响已写正文。
- 用户确认“按影响分析执行大修”。

升级前先把 `追踪/审阅反馈.md` 中本轮约束作为 RIA 的输入，避免新会话丢失用户意见。
