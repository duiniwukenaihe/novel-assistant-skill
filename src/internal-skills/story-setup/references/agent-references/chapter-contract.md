# chapter-contract.md：章节契约

章节契约是每章正文写作前的硬边界。它不替代细纲，而是把细纲压缩成“本章必须交付什么、不能破坏什么、写完如何验收”。

## 何时使用

- `story-long-write` 内部模块在日更续写每章开写前必须使用。
- 细纲缺失时，先补细纲，再生成章节契约。
- 用户要求“直接继续写”时，也必须先生成契约，不得跳过。

## 输入

| 输入 | 用途 |
|---|---|
| 当前卷纲 | 确认本章服务哪个卷目标 |
| 本章细纲 | 提取必须情节点、爽点、钩子 |
| 上一章 State Delta（通常来自 `追踪/上下文.md` 最近 3-5 章摘要，或刚完成的上一章 State Delta 输出） | 继承角色、伏笔、资源和时间线状态 |
| 追踪/伏笔.md | 避免提前兑现或遗忘伏笔 |
| 追踪/角色状态.md | 避免角色状态回退 |
| Character Invariants | 避免动机漂移和认知泄漏 |
| 已接受的细纲质量结果 | 使用 `追踪/workflow/tasks/<workflow-id>/result-packets/detail_outline_review.result.json`；只接受 `pass` 或 `pass_with_advisory`，并要求细纲路径与哈希仍匹配 |

## 输出模板

每章契约必须写入 `追踪/章节契约/第X卷/第{N}章.md`（旧项目可回退 `追踪/章节契约/第{N}章.md`）。**章节契约写入预检**：目录不存在时先创建卷目录，例如 `mkdir -p 追踪/章节契约/第X卷`；再对目标目录运行 `test -w` 并做临时写入测试（写入 `.write-test-$$.md` 后读取并删除）。如果出现 `Error writing file`、`Permission denied`、目录不可写或临时文件读写失败，停止生成契约，输出当前用户、目标目录/父目录 `ls -ld` 和 ownership 修复建议；不得继续生成正文，也不得假装契约已落盘。如果同章契约已存在，覆盖前必须确认这是同一次写作的回炉/重写，或在文件内保留上一版的“变更原因”。

## 生成规则

1. 必须情节点来自本章细纲，不得现场发明主线。
2. “禁止事项”必须包含至少一条与伏笔、设定或角色认知有关的约束。
3. “允许新增”为空时写“无”，表示本章不应新增人物、设定或支线。
4. 章尾期待必须对应下一章，而不是泛泛写“引出后续”。
5. 契约通过并落盘后才写正文。
6. `Plot Drift Gate`、内部模块 `story-review` 和章节回炉必须读取对应章节契约；不得依赖聊天记录中的临时契约。
7. 生成契约时加载 `human-resonance-gate.md`；如果本章是过渡章，也必须写出关系/风险/认知/承诺的“小变化”。
8. 生成机器契约时使用 `node scripts/current-contract-build.js <项目根目录> --chapter N --quality-result <项目内相对路径> --write`。细纲中的 B1/B2/B4 质量约束合并到 `mustInclude`；C4/C5/C6/C7 的记忆候选仅写入 `memoryProjection`，须在既有章节提交事务中落库，绝不直接改写 canonical memory。
9. 新格式细纲同时包含“质量触发”和“呈现与连续性”时必须提供匹配的质量结果；结果缺失、细纲路径不一致或哈希过期会阻断契约。未采用这两个区段的旧格式细纲保留 `warn` 兼容状态，待重审后再升级。
10. 新格式细纲声明“剧情单元合同”后，`current-contract-build.js` 必须把 `plotUnit`、`readerExperience` 和 `terminalReserve` 投影进 `追踪/schema/current-contract.json`；正文只兑现本章回报，不得提前消耗未解锁终局储备。
