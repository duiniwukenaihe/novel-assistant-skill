# novel-assistant

一个入口，持续完成中文网文的规划、写作、审阅、拆文与去 AI 味。

`novel-assistant` 演化自优秀的上游项目 [worldwonderer/oh-story-claudecode](https://github.com/worldwonderer/oh-story-claudecode)。我们保留它的网文专业能力，并针对真实生产中的任务中断、记忆漂移、Token 浪费、结构扩容和模型退化继续完善。

> 当前版本优先把单个短篇从构思到完稿做稳；长篇、审阅、拆文、扫榜、导入和去 AI 味同样可用。

## 三分钟开始

### 安装

```bash
npx skills add https://github.com/duiniwukenaihe/novel-assistant-skill.git --path skills/novel-assistant -y -g
```

只安装到当前项目时去掉 `-g`。

### 使用

进入作品目录，在 Claude Code、Codex 或 ZCode 中调用：

```text
/novel-assistant
```

然后直接说目标：

```text
开始写一本番茄玄幻长篇
开始短篇写作
继续上次未完成任务
审阅当前卷的剧情、人物和钩子
把第一卷从 12 章扩成 20 章
拆解这本书并提炼可吸收技巧
对正文去 AI 味，但不要改剧情
```

新目录只显示新建入口；已有作品先显示未完成任务数量。需要选择时优先使用上下键菜单，无法使用时回退为数字选项，并始终允许直接聊天纠偏。

## 为什么用它

| 真实问题 | novel-assistant 的设计 |
|---|---|
| 子 skill 太多 | 用户只记住 `/novel-assistant`，内部自动路由 |
| 换会话就忘记进度 | Workflow 保存任务、阶段、断点和下一步 |
| 每次重读整本书 | Memory 只召回当前步骤需要的事实、人物和钩子 |
| 从脑洞直接写正文 | 先设定、大纲和 Brief，再逐节/逐章写作 |
| 修改上游后仍沿用旧稿 | 影响链自动标记失效的 Brief、正文和检查结果 |
| 章节扩容后全线错位 | 先审计影响，再迁移标题、内容、钩子和时间线 |
| 去了 AI 词仍不好看 | 机器门查污染，故事门查逻辑、情绪、人物和爆点 |
| 模型重复、工具报错烧 Token | 健康门、重试预算、可信断点和托管早停 |

## 工作流

专业模块负责写作、审阅或拆文；Workflow 负责把它们连成一件能继续、能纠偏、能恢复的任务。

```mermaid
flowchart LR
    A["用户目标"] --> B["定位当前阶段"]
    B --> C["Memory 装配可信事实"]
    C --> D["专业模块只执行当前一步"]
    D --> E["机器门 + 故事门"]
    E --> F{采用吗}
    F -->|修改| B
    F -->|采用| G["提交正文、事实与记忆"]
    G --> H["推荐自然下一步"]
```

小的文字修改留在当前正文；人物动机、因果、反转或节奏发生变化时，先更新设定/大纲，再重新生成受影响的 Brief。不会用下游润色掩盖上游规划错误。

### 短篇

```mermaid
flowchart LR
    A["资讯 / 素材 / 脑洞卡"] --> B["设定与人物"]
    B --> C["节奏与全篇小节大纲"]
    C --> D["当前节 Brief"]
    D --> E["只写当前节"]
    E --> F["双门验收"]
    F --> G{最后一节}
    G -->|否| D
    G -->|是| H["合稿 / 精修 / 发布检查"]
```

扩容、缩容、插节、删节或重排时，先更新全篇规划，再决定哪些 Brief 和正文可以保留。

### 长篇

```mermaid
flowchart LR
    A["故事核心"] --> B["总纲"]
    B --> C["卷纲"]
    C --> D["阶段细纲"]
    D --> E["章节 Brief"]
    E --> F["正文与验收"]
    F --> G["人物 / 伏笔 / 时间线提交"]
    G --> H{阶段完成}
    H -->|否| E
    H -->|是| I["复盘与跨卷交接"]
```

卷内独立编号便于扩容；原章节名和可用内容优先保留，发布时再导出为全书连续编号。

## Memory 与质量

Memory 不是旧聊天堆积，而是可信的创作事实层：

- 只接收已采用正文、已确认设定和可信结果；
- 按作品、任务和目标章节召回，避免重复注入整本书；
- 区分正式事实、建议和污染输出；
- 使用相对资产标识，项目移动或切换工具后仍可接续；
- 存储层可替换，未来可由外部工作台接管。

每个正文单元尽量当场闭环：

1. **规划准入**：目标、阻力、因果、选择和钩子足以落笔。
2. **机器门**：检查重复、工程词泄漏、异常符号、视角与称谓漂移。
3. **故事门**：检查剧情价值、逻辑、人性、情绪变化和大纲兑现。
4. **采用提交**：只有采用版本才更新正式正文与 Memory。

普通段落走轻量门；高潮、反转、人物转折和设定变更才提高检查强度。

## 能做什么

长篇与短篇写作、审阅修复、长短篇拆文、扫榜选题、去 AI 味、旧书导入、封面任务、章节扩容与发布合稿，全部从同一个入口开始。

## 公开版与本地增强

公开版只包含公开模块和公共 Workflow / Memory 内核，短篇由公开 `story-short-write` 承接。本地可加载不进入公开仓库的增强模块；两者遵守相同任务、记忆和质量协议。

公开发布会自动排除私有扩展、个人素材、原文、Demo、日志、内网地址和本机路径，并运行隐私审计与安装验证。
