# 短篇工作流运行边界设计

日期：2026-07-20

## 目标

让公有与私有短篇 skill 共用一条稳定的单作品生产链，同时明确：

- 一个短篇目录只对应一部作品。
- skill 负责当前作品的阶段、断点、连续性、质量门和正式制品。
- `novel-project` 负责跨作品素材库、长期作者偏好、学习进化和多项目管理。
- 私有短篇是公有短篇的增强版，只增加资讯、卡池、组合和个人领域能力，不复制生产内核。

## 不做什么

- skill 不维护跨目录的长期作者画像。
- skill 不把多个短篇作品塞进同一个作品目录。
- skill 不在私有模块中建立第二个全局任务收件箱或记忆系统；但私有模块存在时必须接管短篇业务首屏和后续完整流程，公开短篇仅作缺省兜底。
- skill 不绑定 `novel-project` 的数据库、端口、页面或传输协议。

## 状态权威

### 工作流任务

`追踪/workflow/tasks/<workflow_id>/task.json` 是阶段、任务状态和恢复位置的唯一权威记录。

`追踪/workflow/current-task.json` 只保存界面焦点。所有写入命令必须显式接收 `workflow_id`，不得从焦点指针猜测写入目标。

### 当前短篇作品

`追踪/private-short-extension/project-state.json` 是当前目录内唯一作品的投影，至少记录：

- `project_id`
- `project_title`
- `active_write_workflow_id`
- `plan_revision`
- `planned_sections`
- `current_section_index`
- `accepted_sections`
- `status`

它不保存跨作品学习。小节事实仍以已接受的小节事务和锚点为准，`project-state.json` 可由这些事实重建。

### 规划与标题

`素材卡.md`、`设定.md`、`小节大纲.md`、标题锁和 Brief 必须绑定当前 `project_id` 与 `plan_revision`。

大纲或标题变化后递增 `plan_revision`。旧 Brief 只标记过期，不静默覆盖；重新生成当前节 Brief 后才能继续正文。

## 启动与路由

环境检查、未完成任务发现和任务选择只经过一次全局 `workflow-entry-guard` 与任务收件箱。它们不负责展示短篇素材、选题或创作菜单。

全局入口把任务路由到短篇后，若私有增强存在，私有短篇模块必须接管作者看到的第一个短篇业务首屏和后续全部短篇阶段；公有短篇不得再显示另一套菜单。这里的“只显示一次”指不重复全局任务收件箱，不是取消私有业务首屏。

私有短篇业务首屏根据项目状态返回：

- 新项目：资讯发现、保留素材池、已有素材开写、审阅或导入。
- 活动项目：首项固定为“继续当前阶段（推荐）”，并展示作品名、已采用小节和最后可信断点。
- 可恢复项目：先从项目状态、标题锁、Brief 和采用锚点重建唯一任务。
- 已完稿项目：只提供审阅、回炉和导出；新故事必须创建新目录。

公有与私有流程都使用 `short_write`。任务快照通过 `workflow_profile` 和 `workflow_owner` 固定实际 owner，恢复时不得静默从私有降级为公有。

## 项目内运行记忆

skill 只保留完成当前作品所需的运行记忆：

- 已确认人物与称谓
- 已发生剧情事实
- 当前人物状态
- 已回收和未回收钩子
- 上一节承接锚点
- 当前作品风格卡与禁写项
- 用户对当前作品的已确认修改

这些内容来自正式规划制品、已接受小节事务和用户确认，不依赖聊天 recap。

正常短篇生产不再把跨作品学习建议写入 skill 的全局记忆。`memory_updates` 仅保留兼容能力，不作为短篇完成条件。

## novel-project 集成边界

skill 在关键接受节点生成通用结构化出站事件，供任何外部管理器选择消费：

- `material_accepted`
- `setting_accepted`
- `outline_accepted`
- `section_accepted`
- `user_feedback_accepted`
- `story_completed`

事件只包含当前作品标识、制品路径、摘要、证据哈希、时间和可选标签，不包含私有素材库实现或数据库信息。skill 将事件写入项目内 outbox；是否吸收到长期记忆由 `novel-project` 决定。

详细消费契约见 [novel-project-outbox-contract.md](novel-project-outbox-contract.md)。这一仓库只生产事件，不实现 `novel-project` 的数据库、页面、同步任务或反向写回。

## 上下文策略

短篇正文阶段使用一个合并后的最小上下文包：

1. 当前 Brief。
2. 当前设定与小节大纲的必要摘要。
3. 上一节采用锚点和末尾承接片段。
4. 当前作品风格卡与禁写项。
5. 与当前节直接相关的已接受事实。

短篇正文、修订和质量门不再额外加载固定 3600 token 的独立全局记忆包。规划、资讯学习和审阅阶段可按需装配更宽上下文，但仍按当前 `project_id` 隔离。

## 兼容策略

旧项目首次进入时只补齐缺失的项目身份和规划版本字段，不改正文、设定、大纲或已接受小节。

若一个旧目录检测到多个独立作品身份，停止自动迁移并提示拆分目录；不得自动合并。

若私有 registry 不存在：

- 新任务使用公有短篇流程。
- 已固定为私有 owner 的任务保持阻塞，等待恢复私有模块，不能静默切换。

## 验收标准

1. 两个短篇任务切换焦点时，标题确认和任何写入都只影响显式 `workflow_id`。
2. 调用 `/novel-assistant` 只出现一次启动/恢复菜单。
3. 新短篇在 `project_seed` 后一定存在完整项目身份。
4. 大纲变更会让旧 Brief 过期，并阻止直接写正文。
5. 正文阶段只要求读取一个最小上下文包。
6. 接受素材、规划、小节和全文时会生成可重放的 outbox 事件。
7. 没有 `novel-project` 时，skill 仍能独立完成当前短篇。
8. 私有和公有流程继续共享同一逐节事务、质量门、采用锚点和全文组装内核。
