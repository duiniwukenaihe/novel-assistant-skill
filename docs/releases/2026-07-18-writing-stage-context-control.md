# 2026-07-18 写作阶段上下文与单命令门禁计划

## 目标

解决短篇逐节写作中反复读取 skill、协议、脚本和历史回执导致的会话膨胀，并把同一控制方式扩展到长篇当前章节生产。公开短篇与本地私有短篇共享生产内核；私有模块只增加资讯、素材、个性化方法和本地记忆，不复制或绕过公开质量流程。

## 实施计划与状态

- [x] 修复私有短篇旧协议：机器门和故事门统一使用单命令执行器，不再要求模型逐个调用检查器。
- [x] 为 Brief、正文、机器门、故事门、反馈影响和采用锚点生成阶段最小上下文包。
- [x] 新增 Brief 收束命令：校验 Brief 完整性、绑定依赖快照并推进状态。
- [x] 新增采用锚点命令：复用已经通过的门禁证据，更新当前节锚点与下一节交接，不重复审阅。
- [x] 为长篇章节 Brief、正文、正文验收和章节提交准备最小上下文包。
- [x] 将长篇正文验收收束为机器门与八维故事质量门两条命令；失败只返回当前章修订。
- [x] runner 同时支持短篇和长篇阶段包；宿主只收到当前写作单元的文件白名单与预算。
- [x] 写作阶段平台源码读取预算设为 0；受控生产命令列入安全白名单。
- [x] 将新增运行脚本纳入单目录 bundle、setup safe bash guard 和三端本地安装。
- [x] 运行语法、JSON、diff 与针对性确定性回归。
- [ ] 由作者在新 Claude Code 会话中做真实短篇逐节试写；旧会话已经产生的缓存上下文不会被 skill 反向清除。

## 短篇生产路径

```text
全篇规划
-> 当前节 Brief
-> Brief 收束命令
-> 只写当前节候选稿
-> 单命令机器门
-> 一次故事质量判断
-> 采用锚点命令
-> 下一节 Brief
```

补写或改写会使本节旧门禁失效，但不会重新读取整份 skill 或重复扫描全篇。故事质量门与机器门各执行一次；只有 blocking 才返回当前节修订。

## 长篇章节路径

```text
章节 Brief
-> Brief 复核
-> 当前章候选稿
-> 单命令机器门
-> 八维故事质量门
-> chapter commit
-> 记忆与交接投影
```

当前实现控制的是单章生产上下文。卷纲、阶段细纲、扩容和全书审阅仍使用各自 workflow，不应被塞进单章正文会话。

## 运行边界

- runner 托管模式可以对子进程实施预算、心跳、健康中止与一次受控重试。
- Claude Code、Codex、ZCode 已打开的交互会话属于协作模式。skill 可以限制后续读文件和工具调用，但不能删除宿主已经缓存的上下文，也不能拦截隐藏思考。
- 本轮不把固定 token 数写进创作规则。阶段包按资产类型和阶段动态压缩，长篇当前章默认预算只是运行上限，不是正文长度限制。
- 质量门阻止污染和明显违约，不代替作者对故事吸引力、人物情感和平台适配的最终判断。
