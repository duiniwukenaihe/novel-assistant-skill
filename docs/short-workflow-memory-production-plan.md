# 短篇工作流与记忆生产化计划

## 目标

让公开短篇模块与私有增强模块共用同一套可靠的生产内核：每一节从 Brief、候选稿、质量门到正式采用均有唯一事实来源；项目可移动、可跨 Claude Code / Codex / ZCode 恢复；任何失败都停在最后可信断点，不猜节数、不重复写作、不污染后续上下文。

## 核心约束

1. `current-task.json` 只表示界面焦点，任务事实只从 `workflow_id -> task.json` 读取。
2. 正式小节采用不可变文件 `正文/第NNN节.md`；`正文.md` 仅是发布合稿，不承担工作流事实来源。
3. 一次采用必须原子完成：候选稿校验 -> 质量门证明 -> 小节提交 -> 记忆投影 -> 工作流回执 -> 项目状态推进。
4. 下一节上下文只读已采用小节、结构大纲、当前 Brief 和结构化记忆，不读取整篇累计正文。
5. 私有短篇是公开短篇的增强层，只扩展资讯、素材、学习和个性化阶段，不复制生产阶段状态机。
6. 所有菜单使用稳定数字选项；推荐项标注“推荐”，成功动作自动进入其承诺的下一阶段。

## 实施阶段

### P0 小节事实与事务

- [x] 新增小节提交事务，将正式稿写入 `正文/第NNN节.md`。
- [x] 采用锚点记录 `section_commit_id`、不可变路径、内容哈希和任务尝试。
- [x] 采用失败不得提前推进 `project-state.json`。
- [x] 采用证明只验证对应小节，不再验证整个 `正文.md`。

### P0 下一节上下文

- [x] 从上一节采用锚点读取真实结尾、人物状态、已揭示信息、未回收钩子和下一节交接。
- [x] Context Packet 按 `section + attempt` 版本化，禁止覆盖历史上下文。
- [x] 缺少结构化采用元数据时停在“补验收元数据”，不得自动猜写下一节。

### P0 全文收束

- [x] 新增确定性合稿器，按已确认小节计划检查 `1..N` 连续性。
- [x] 检测缺节、重复标题、陈旧哈希和计划外第 `N+1` 节。
- [x] 全部通过后生成 `正文.md`，否则返回可操作的数字菜单。
- [x] 合稿后新增全篇总编辑验收，检查开篇信息负载、小节功能/篇幅曲线、重要人物主动性、阻力动机、主角身份效用、高潮跑道、结尾后果和标题兑现。
- [x] 总编辑 `revise` 不直接改稿，而是按 `editorial_review -> feedback_impact_sync -> accepted_plan -> 规划回写 -> Brief/正文复检` 回流。
- [x] 去 AI 味与最终发布门必须持有当前合稿的总编辑回执；内部 `pass` 对作者显示为“故事层可进入表达清理”，陈旧审阅不能发布。
- [x] 去 AI 暂存稿提交前执行删损保真门：小幅删冗正常，显著缩短只补回剧情功能、人物反应、行动后果和承接，不机械补字；最终发布门验证保真回执。
- [x] 总小节锁定同时校验总节数、全篇目标篇幅、发布形态、逐节结构功能与终局完成边界；缺项返回补规划菜单，不进入正文。
- [x] 结构修改后自动生成影响审计，区分保留、失效、重算与复检资产，并把每项绑定回责任阶段。
- [x] 公有 `hook_value_gate` 与私有 `hook_retention_gate` 共用钩子价值证据卡；未验证标题承诺、开篇压力、剧情峰值、主角主动性和因果链时不得进入 Brief。
- [x] 必需阶段没有完成脚本或已装配上下文时不再进入假运行状态，统一返回四项恢复菜单。

### P1 记忆与多宿主恢复

- [x] 正式采用时投影人物、因果、钩子、用户确认和风格事实。
- [x] `workflow-state-machine apply-result` 成为公共记忆提交入口，runner 不能拥有独占逻辑。
- [x] 记忆证据使用项目相对路径和 `workflow_id / stage_attempt_id / section_commit_id`。
- [x] Claude Code、Codex、ZCode 切换后从同一任务事实恢复。

### P0 反馈决策与完整记忆（2026-07-22 补强）

- [x] 对话意见只进入任务工作记忆，不直接成为作品事实。
- [x] 用户确认助手总结后形成 `accepted_plan`，绑定方案、影响小节和资产回写顺序。
- [x] 规划回写验收后再投影 `planning-constraints`，避免 memory 先于大纲生效。
- [x] 每节采用事务补齐人物关系、关键选择和因果推进，继续保留人物状态与钩子。
- [x] 受影响 Brief 与正文复检由 revision queue 自动生成，不要求作者逐项提醒。

### P1 公私有统一

- [x] 公开与私有工作流共享 `Brief -> 写作 -> 双门 -> 采用 -> 下一节/合稿` 生产子图。
- [x] 私有模块只注入资讯抓取、素材池、血缘、学习记录和个性化策略。
- [x] 发布构建继续使用既有公开清洗门；本轮没有改变私有资产边界。

### P0 Workflow / Memory / 子模块边界收束（2026-07-22）

- [x] `accepted_plan` 保存最终确认方案；规划记忆只投影该方案，不投影原始聊天碎片。
- [x] V2 Result Packet 在接受前校验当前 Memory Revision，陈旧上下文不得提交。
- [x] 专业子模块只返回 `memory_updates` 建议，禁止直接调用记忆写入器。
- [x] 记忆领域防火墙隔离 workflow、路由、菜单、安装与会话控制信息，禁止进入故事 canon。
- [x] V2 Result Packet 必须声明与当前阶段一致的 `owner_module`，公有/私有模块不能靠缺省串线。
- [x] 误分类反馈通过幂等状态机命令修正审计，保留历史且不手改任务 JSON。
- [x] 对真实果汁任务执行反馈重分类，并把已确认方案绑定到受控规划回写阶段；正式创作资产保持未修改，等待作者实际执行。
- [x] 构建私有 bundle，同步 Claude Code / Codex / ZCode 本地安装。

全局 workflow/memory 不变量、跨专业模块边界与宿主能力限制见 [Workflow / Memory 全局生产架构计划](workflow-memory-architecture-plan.md)。本文件只保留短篇生产子图的实现记录。

### 验证

- [x] 单测：采用事务、幂等恢复、失败不推进、上下文只读上一节。
- [x] 集成：九节短篇不会生成第十节；移动项目后可恢复；重复会话不能重复采用。
- [x] 回归：公开短篇与私有短篇使用相同生产内核，现有门禁与菜单测试保持通过。
- [x] 真实宿主测试由用户按需执行，不在日常修复中自动消耗付费模型 token。
- [x] 聚焦回归覆盖：结构证据包、审阅卡完整性、公私有路由、缺卡停靠、revise 回流与任务级 `editorial_review` 工作记忆。

## 验证记录

- 目标回归：短篇事务、标题锁、Brief、正文、修订、状态机与宿主适配共 `144/144` 通过。
- `new-book` 只读复制验证：旧九节迁移为不可变小节，预检不修改 `正文.md`，受控合稿后进入私有 `short_deslop`，任务中不存在第 10 节推荐。
- 旧版兼容：仅在旧锚点同时保留机器门与故事质量门通过证据时迁移；缺少证据则阻塞，不生成正文补洞。

## 完成判定

只有当小节采用、下一节上下文、全文合稿、记忆投影和跨宿主恢复都以同一任务事实为准，且失败路径不会推进节数，短篇工作流才可标记为生产可用。
